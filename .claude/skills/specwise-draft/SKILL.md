# Draft: Quick Spec Draft

## What This Command Is

Draft is the "stenographer" mode of the Specwise system. It serves a specific user need: **"I know what I want. Don't ask me questions. Just show me a structure and I'll edit it myself."**

The difference from `/sw:spec`: spec does complexity assessment, proactive clarification, multi-round guidance; draft skips all of that and directly generates a complete Spec draft.

Key point: **Draft skips "clarification," not "Spec."** The output is still a complete 3-layer Spec — the system just doesn't do proactive guidance, leaving all judgment to the user.

## Workflow

### Step 1: Receive Task Description

Receive the task description passed by the user via `/sw:draft`. No complexity assessment, no flow recommendations.

### Step 2: Generate Complete Spec Directly

Based on the user's description, generate a complete 3-layer Spec:

**Core Layer** (required):
- Objective
- Expected Output
- Constraints
- Definition of Done

**Context Layer** (fill as much as possible):
- Inputs
- Background
- References
- Stated Assumptions — mark all model inferences with ⚠️

**Refinement Layer** (fill if inferable from description):
- Priorities
- Exclusions
- Risks
- Quality Preferences

Because there's no clarification step, the "Stated Assumptions" field will be longer than in `/sw:spec` mode — the model will have more dimensions it needs to infer on its own. Every assumption must be explicitly marked with ⚠️.

### Step 3: Add Disclaimer

Append to the end of the Spec:

> ⚠️ **This Spec was generated without a clarification process and may contain uncovered requirements or inaccurate assumptions. Please review carefully and modify before confirming.**

### Step 4: Wait for User Modifications and Confirmation

The user can:
- Confirm directly → enters executable state
- Modify then confirm → system updates the Spec based on user changes
- Request additional clarification → system can ask questions about specific parts

### Step 5: Consistency Check (warn but don't block)

When the user modifies the Spec and prepares to confirm, the system performs a quick consistency check:

- Do the objective and constraints contradict each other?
- Does the definition of done match the expected output?
- Do priorities conflict with exclusions?

If inconsistencies are found, **warn the user but don't block confirmation**:

> "I noticed a potential inconsistency: [specific description]. You can choose to modify it, or keep it as-is and confirm."

### Step 6: Guide to Execution After Confirmation

After confirmation, prompt:
> "Spec confirmed. Use `/sw:execute` to execute based on this Spec."

## Relationship with Other Commands

Draft is the fastest of the three paths:

```
explore (slowest, most thorough) → spec (standard flow) → draft (fastest)
         ↘                           ↓                    ↙
                           Confirmed Spec
                               ↓
                            execute
```

Users can switch at any time: if the Spec generated by draft isn't good enough, use `/sw:spec` to go through the full process.
